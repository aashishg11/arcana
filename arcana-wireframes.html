✦Arcana wireframes · conv.4

How to read these

Eight frames. The first five are prioritized by how much of Google's ai-samples surface each exercises. The last three — Camera, Onboarding, Import — carry no AI but are load-bearing: they're how a real collection gets into the app, which is the prerequisite for everything the AI screens do. Medium fidelity: enough structure to judge hierarchy, annotations beside each phone for behavior and the capability each region demonstrates. Gold-bordered notes are open questions for the pixel pass. Theme toggle is top-right; both modes are designed, not one derived from the other.

Still excluded — genuinely commodity: Category (browse list), Edit Details (gap-fill form), Settings (status readout). Prose in SCREENS.md covers these; a wireframe wouldn't surface anything new.

Token-by-token, no flash on settle

NFT BADGE

Provenance label, not state

The gold NFT Redemption Pop badge is the one place gold appears — reserved for prestige. It's a catalog fact (isNftRedeemable), purely a display badge. There is no redemption state machine; all these Pops are physical.

SNAPSHOT vs REFRESH

The naming is load-bearing

The market section is already live-on-render, so there's no "Refresh price" button. Snapshot today's price commits a ValueSnapshotEntity to tracked history — a different action. Debounced: same-day within 5% surfaces "Already up to date" instead of a duplicate row.

VALUE HISTORY

Honest about thin data

From value_snapshots, 90-day default. Under two snapshots it reads "Price tracking started — first sync this Sunday" rather than drawing a misleading flat line on an empty axis.

LISTING WRITER genai-writing · P2

Where the eighth capability lands later

Post-MVP, the overflow menu gains "Draft a listing" — genai-writing-assistance rewrites the item's metadata into sale copy. Not v1, but Detail is its home, which is why it's flagged here rather than drawn.

Open: does the listing writer pull condition + market context into the draft, or just identity fields? The former is more useful and exercises more of the data model.

Camera

capture/camera · state: CameraReady

Entry to the capture flow — new acquisitions only, not re-cataloging. The screen's whole job is to capture one frame; the AI runs afterward on Review. The barcode fallback lives here, deliberately understated: it's the demoted path, because the image cascade is the showcase, not barcode scanning.

Scan barcode
BARCODE FALLBACK

Present, deliberately quiet

Bottom-left, low contrast, no badge or pulse. It's always available for users who know they want it, but it never competes with the shutter. This visual weight is the architectural decision — barcode was demoted from a co-equal cascade to a UI fallback so the image-first pipeline stays the star.

Tapping it decodes the barcode and jumps the cascade straight to Stage 4 (catalog lookup by UPC), skipping segmentation/classification/OCR.

SHUTTER

Tap to capture, nothing more

Single tap freezes the frame and starts the cascade. Press-and-hold was considered and rejected as over-engineered — there's no second gesture to disambiguate, so a long-press would only add a thing to explain.

NO FRAMING OVERLAY

No "center your Pop here" guidance

The viewfinder stays clean. Framing reticles and instructional text are condescending and unnecessary — the segmentation stage handles messy framing. The absence is a decision, not an omission.

PERMISSION STATE

Denied → rationale, not a dead end

If camera permission is denied: "Arcana needs camera access to identify new items," with Grant permission and Not now. v1 doesn't handle permanent denial in-app — it points the user to system settings.

TWO EQUAL PATHS

Neither is the primary CTA

Both rendered as equal-weight outlined cards — no iris-filled button forcing a choice. Start fresh comes first so the free path isn't buried under a paywall the user may not own. The import label is specific ("from HobbyDB"), not vague ("existing collection").

THE BOOTSTRAP

Why this screen is load-bearing

Import is the only way the 283-item, $22,634 collection lands in the app — which is what makes the portfolio value real, the RAG answers grounded, and the "you already own two" callout possible. No AI here, but everything downstream depends on it.

HOW THE FILE ARRIVES

SAF picker + share intent

Import opens Android's Storage Access Framework (ACTION_OPEN_DOCUMENT) for CSV mime types. Arcana also registers a share-intent filter, so a user can share a CSV email attachment straight into the app.

Open: the three-screen "export from HobbyDB" walkthrough — does it appear before the picker, or only if the picker returns nothing? Pre-showing it helps first-timers but adds friction for the prepared.

BLOCKS NAVIGATION

No back gesture mid-import

Determinate progress (row index / total) — not a spinner, because the count is knowable and seeing "47 of 283" reassures. Back is disabled during Parsing and Writing; it re-enables only on a terminal state (Complete, PartialComplete, Failed). The copy says why.

THE REAL DATA WORK

This is where the CSV quirks get handled

Behind this bar, HobbyDbCsvImporter strips =HYPERLINK() wrappers, splits comma-separated Series and Production Status into lists, casts UPC to String to keep leading zeros, and flags NFT Redeemable. The integration test asserts 283 rows and 142 NFT items against the real export.

PARTIAL COMPLETE edge case

Lenient with warnings, user in charge

A bad row logs a warning and is skipped — the rest succeed. The terminal state reads "280 items added, 3 skipped" with Review skipped items and Continue anyway. Strict all-or-nothing is wrong for a personal collection; so is polluting the DB with half-parsed rows.

Open: wireframe the Complete and PartialComplete terminal states as their own frames, or is the Writing state plus this rule enough for the pixel pass?

ONE-SHOT

Import and forget

After this, the user is the source of truth. Re-import (Settings → "Import another CSV") is append-only with no merge logic — duplicates are the user's to resolve. Deliberate v1 simplification, not a gap.
